## 2026-07-14 (cont.) — Buyers · Sellers · Past orders; 659-product tender coverage; data currency

### 1. Tenders were matching the wrong notices (BUG L-004)
Owner: *"the product is tea but I see many buyers — do you guarantee those are buying tea?"* **No.**
A CPV search matches a notice if the code appears ANYWHERE in it, so a school's 100-item food
framework (which lists tea among the lots) came back under "tea" — the feed showed buyers of *Bread*
and *Mineral water*. Every notice is now classified against the searched CPV: `contract` (its main CPV
is this product) · `lot` (a lot's main CPV is) · `basket` (buried line item). **Baskets dropped.**
1,574 scraped -> 923 basket / 343 lot / 308 contract; shown drops 670 -> 269. Wood pellets = 85 whole
contracts; tea = 0 contracts, 17 real lots. For a lot, TED's headline describes the WHOLE contract, so
the row leads with the product and shows the contract it sits inside.

### 2. Sellers exist — as won contracts (ADR-0005)
Owner: *"sellers do the business on their own way — not everybody goes outside and tells 'I am selling
this'."* Correct, and it settles the design: **no public feed of sellers offering goods exists**, so a
seller list cannot be collected — it must be DERIVED from a public event. TED award ("result") notices
name `winner-name`, `winner-country`, `winner-decision-date` and the value. So:
- **Past orders** = the awards (both sides named, value where TED gives one unambiguous figure).
- **Sellers** = those awards aggregated by winner (contracts won, last win, who they sold to).
- **Buyers** = open tenders. Three questions, three tabs.
TED also exposes `winner-email` / `winner-person` — never stored, never shown (Golden Rule).
Live: 21,075 award rows -> 10,450 on-product -> 8,111 sellers.

### 3. Coverage: 18 -> 659 products (BUG L-005)
No official HS<->CPV crosswalk exists, so `reference/_gen_cpv.py` builds one: propose a CPV from the HS
heading's text, then **verify live against TED** — a code survives only if TED files on-product notices
under it. 810 proposed -> **654 verified**, 156 dropped as fiction; + the 18 hand-checked pilots = 659.
Two guards, both learned from failures: an HS-chapter -> CPV-division rule (without it "Cashew nuts"
matched CPV *fastener* nuts, "Tea" matched "Flavoured yoghurt") and a 0.6 score floor (at 0.5 "Crude
oil" matched "Oil paints"). A verified match can still be a neighbour ("Vegetables, dried" -> CPV
"Frozen vegetables"), so `cpv-match.json` ships the matched label and the UI prints it per product.
Final pull: **6,006 open tenders / 641 products with sellers + past orders**.

### 4. Data currency — the owner was right, and it was worse than it looked
*"The newest data is 2024 but we are in 2026."* BACI (the bulk history backbone) **ends at 2023**, and
the Comtrade API had only ever covered **13 products** before its quota throttled — so 1,227 of 1,240
products were frozen at 2023. Fixing it exposed two real bugs:
- **Batching never fired.** `run_multi` called every API source ONE PRODUCT PER CALL, so the
  comma-separated `cmdCode` batching added earlier was dead code: 2,480 single-product calls, ~1
  product/minute, **17 hours**. Batched sources now receive the whole chunk (`ComtradeSource.batched`).
- **We downloaded 30x the data we threw away.** Comtrade returns every customs-procedure x
  transport-mode breakdown — 38,200 rows per call, of which `_is_total_row` keeps 1,272 and discards
  the rest *client-side, after paying to download them*. Asking the server for the aggregated cell
  (`customsCode=C00, motCode=0, partner2Code=0`) returns EXACTLY the same cells — verified 1,272 vs
  1,272, **0 missing, 0 extra, 0 value disagreements**. 35s -> 5s per call.
Result: **48 products/minute**; full refresh ~25 min. **All 1,240 products now reach 2025.**
Flows 821k -> **1.9M**; signals 9.7k -> **462k**.
