## Core types (current)
- **`trade_flows`** (`etl/tradepulse_etl/db.py`) — IMPLEMENTED (SQLite). The Layer-1 fact table.
  PK(reporter,partner,hs6,period,flow); **freq('A'/'Q'/'M')** + value_usd + quantity + source +
  published_date. Grain lives in `period` (YYYY | YYYY-Qn | YYYYMM) so monthly/quarterly/annual rows
  coexist (UI toggle); `freq` labels it. Populated by `pipeline.run_multi` from ONE OR MORE sources,
  passed through **`merge.merge_flows`** (dedupe: one row per cell — national authority > freshness >
  priority; never sums two sources). `source`/`published_date` = the winning source's freshness stamp.
- **`signals`** (`etl/tradepulse_etl/db.py`) — IMPLEMENTED. Filled by `signals.compute_signals`
  (PURE, deterministic). One row per (reporter,hs6,flow,period) clearing the §6 floors; band ∈
  {minor,moderate,significant,surge,collapse,new}.
- **`companies`** (Layer 2) — IMPLEMENTED as curated JSON content: `content/companies/pellets.json`
  (id, name, role, country, hs6[], profile_url, evidence_source, evidence_url, verified_date;
  **no contact fields** — Golden Rule). Read by `web/app/lib/companies.js`. Moves to a table later.
- **`requirement_pages` / `requirement_items`** (Layer 3) — IMPLEMENTED as curated JSON:
  `content/requirements/pellets-{jp,kr,eu}.json` (§8 template + change_log). `web/app/lib/requirements.js`
  DROPS any item missing source_url+verified_date. Git history = the change-log/audit trail.
- **`watches` / `alerts` / `locked_page_clicks`** (Layer alert engine) — IMPLEMENTED as append-only
  NDJSON capture (`data/watches.ndjson`, `data/alerts.ndjson`, `data/locked_clicks.ndjson`, gitignored).
  Alert logic is PURE in `etl/tradepulse_etl/alerts.py`. Move to tables + real email/Zalo delivery later.
- **`tenders`** (Layer 2 / forward demand) — IMPLEMENTED. One row per (notice, product) from EU TED.
  PK(id,hs6); cpv, **match_kind**, title, buyer, buyer_country(ISO3), published, deadline, url.
  `match_kind` is the on-product test — a CPV search matches a notice if the code appears ANYWHERE in
  it, including as one buried line item of a mixed contract:
  `contract` = the notice's MAIN cpv is this product · `lot` = a LOT's main cpv is this product ·
  `basket` = the code only appears in the full list. **Baskets are dropped at export** (a school food
  framework listing tea among 100 items is not a tea lead). Buyer ORGANISATION only — never a contact.
- **`awards`** (Layer 2 / PAST ORDERS, and the evidence base for SELLERS) — IMPLEMENTED. One row per
  (notice, product, winner) from TED award ("result") notices. PK(id,hs6,winner); winner,
  winner_country, buyer, buyer_country, award_date, value, currency, cpv, match_kind, url.
  **Sellers are derived from this table, not collected** (ADR-0005): sellers never advertise, so the
  only public record that a company sells a product is a contract it WON. `value` is stored ONLY when
  TED reports a single unambiguous total — multi-lot notices report per-lot figures that cannot be
  attributed to a winner, so we carry NULL rather than a wrong number. TED also exposes
  `winner-email` / `winner-person`; we never store or display them (Golden Rule).
- **`reference/cpv_by_hs.json`** — GENERATED, committed, reviewable. HS4 -> the CPV code TED files that
  product under. No official HS<->CPV crosswalk exists, so `reference/_gen_cpv.py` proposes a match
  from the HS heading's text and then VERIFIES it live against TED; a code is kept only if TED really
  files on-product notices under it (810 proposed -> 654 kept). `config.TENDER_CPV` = this map, with
  the 18 hand-checked pilot products overriding it. Match quality ships to the UI (see below).
- **`registry_sellers`** (Layer 2 / real SELLERS — ADR-0006) — IMPLEMENTED. A seller is a company
  APPROVED to export a product (an official registry), NOT a contract winner (that is `awards`). One
  row per (source, approval_no, seller, seller_code): seller, seller_iso, seller_code(M49), approval_no,
  activity, city, section, source_url, verified_date. Filled by `sources/registry.py` (Phase 1: EU DG
  SANTE, keyless, animal-origin -> seafood + honey). A registry SECTION maps to the HS products it
  covers via `config.SELLER_SECTIONS`; `export.build_sellers_web(hs)` returns the deduped exporters for
  a product. Golden Rule: org + approval + public source + verified date; never a contact person.
- Remaining tables below are still PLANNED.

### Web snapshot contract (`snapshot-<hs>.json`, generated by `export.py`)
**Country-centric (ADR-0003), and SLIM** — 1,240 products have to ship, so the file is written with
short keys and shared indexes, then REHYDRATED by `web/app/lib/snapshot.js` into the shape components
render (no component knows the file is slim). Shape:
`{ generated_at, hs6, product{name_en,name_vi}, latest_period, is_sample, sources[],
   periods{ A:[…], Q:[…] },              // the shared period index, ONE per grain per file
   countries[ { c: code,
                e|i: { v: value_usd, p: period, f: freq, y: yoy_delta, b: band, d: direction,
                       h: [values…],     // aligned to periods[f]; null = no figure that period
                       bf: { <freq>: {…slot…} } }   // NON-default grains only, never a copy of default
              } ] }`
Country NAMES are not in the file — they live once in `countries.json` (they used to repeat in all
1,240 files). No `feed[]`: the web derives the signal feed from `countries` at the chosen grain.
Cost of the slimming: 310KB -> ~64KB per product; **all 1,240 ship (~79MB)**, export runs in ~96s.

### Tender / award / seller contracts (generated by `export.py`, read by `web/app/lib/tenders.js`)
- `tenders-<hs>.json` — `[{id, hs6, title, buyer, buyer_country, buyer_code(M49), match, cpv,
  deadline, published, url}]`. **Still-open only** (TED's "ACTIVE" scope means *published*, not open,
  so passed deadlines are dropped) and **on-product only** (`match` ∈ contract|lot; baskets dropped).
- `awards-<hs>.json` — `[{id, hs6, title, buyer, buyer_country, buyer_code, seller, seller_country,
  seller_code, match, cpv, date, value, currency, url}]` = PAST ORDERS.
- `sellers-<hs>.json` — **real exporters from approval registries** (ADR-0006), NOT award-derived:
  `[{seller, seller_country(ISO2), seller_code(M49), approval_no, activity, city, source, url,
  verified}]`. Empty for products no registry covers yet (honest "coming soon"). (Superseded shape,
  award-derived, was: `[{seller, seller_country, seller_code, wins, last,
  buyers[], value, currency, url}]`, ranked by wins then recency. Totals sum only within ONE currency;
  mixed currencies -> `value: null`.)
- **`-TOTAL` files** are the ROLLUP for "All products": every product's rows, deduped (one notice can
  match an HS4 heading AND its HS6 children, which share a CPV — 6,006 raw -> 2,687 unique open
  tenders). Each row keeps its `hs6` so the aggregate view can name the product per line.
- `cpv-match.json` — `{ <hs>: {cpv, label, exact} }`. `exact: true` = hand-checked pilot mapping;
  `false` = generated + TED-verified, so the match is the right domain but not always the identical
  good ("Vegetables, dried" -> CPV "Frozen vegetables"). The UI PRINTS this per product — broad
  coverage is only honest if the user can see what it matched.

### Sourcing contract (`sourcing-<hs>.json`, generated by `sourcing.py` — quarterly, focus reporters)
`{ "<reporterCode>": { export|import: { periods[], partners[{code,name_en,name_vi,value_usd,share,
yoy_delta,direction}], series[{code,name_en,name_vi,values[]}] } | null } }`. Focus reporters =
`config.FOCUS_REPORTERS`. Consumed by `web/app/country/[code]/page.js` (PartnerTable + SourcingChart);
absent reporters fall back to the annual history sparkline.

## Signal computation (deterministic — plan §6)
- Granularity: reporter country × HS-6 × quarter, split by flow.
- Comparison: **YoY only** (same quarter last year) — never QoQ (seasonal products).
- `delta = (value_this − value_same_period_last_year) / value_same_period_last_year`.
- Noise floors (all must pass): value ≥ $10M this qtr; base ≥ $2M; ≥ 4 quarters history.
- Bands: ±15–30 moderate · ±30–60 significant · beyond ±60 surge/collapse · base≈0 & now ≥$5M = new lane.
- Suppress −15…+15% ("minor" excluded by design). Declines are signals too.
